# 第三次架构改动思考：多轮上下文感知与持久化会话管理

## 1. 核心改动背景与痛点分析（Why）

在之前的第二次改动中，系统已经通过 **Hierarchical（父子切片回溯）**  和 **Rerank（交叉注意力重排）**  实现了在金融高壁垒知识库中“精准检索”和“严谨回答”的能力。然而，系统在交互范式上仍停留在**单轮独立检索**阶段，暴露出以下三个核心痛点：

- **指代消解失败（Coreference Issue）** ：金融风控专家在实际业务分析中，倾向于进行连续递进式追问（例如，先问“什么是Downturn LGD？”，再追问“那金融机构在量化**它**时面临什么挑战？”）。在单轮机制下，检索器无法理解“它”指代上文，导致检索完全脱靶。
- **历史上下文污染与Token浪费**：若粗暴地将前文所有聊天记录直接拼接作为大模型的 Prompt 输入，既会迅速刺破大模型的 Token 窗口限制，又会引入大量无意义的噪声（如系统初始化欢迎语），干扰向量数据库的相似度匹配。
- **状态管理脆弱与脏数据入库**：在 Streamlit 前端渲染中，若采取“先入库再流式生成”的常规逻辑，一旦遇到网络抖动、API 报 502 错误或触发风控底线拒答，未完成的脏数据就会永远残留在用户的会话历史中。

## 2. 技术演进与核心重构方案（What & How）

为了在工程健壮性与业务深度之间取得平衡，本次改动核心落实了“三位一体”的重构策略：

### 🛠️ 核心重构 A：后端引入“历史感知检索器”（History-Aware Retriever）

- **重构逻辑**：在 `test.py` 中，放弃了直接将原始 `retriever` 塞入组合链的传统做法。在初始化底层大模型驱动（DeepSeek）之后，引入了 `create_history_aware_retriever`。
- **机制流转**：

  1. 系统接收到用户带有指代的全新追问（如：“如果不考虑*这种联动效应*会怎么样？”）。
  2. 包装后的检索器并**不直接去查向量库**，而是先利用一个独立的轻量级 Prompt（`contextualize_q_prompt`）配合大模型，结合传入的 `chat_history` 进行 **Query 重写（Query Condensation）** 。
  3. 将模糊问题完美消解指代，改写为独立的长句 Query（如：“在计算监管资本时如果不考虑PD与LGD的联动效应会导致什么后果？”）。
  4. 最终，拿着这个干净、饱满的独立 Query 去调用底层的父子切片和 Flashrank 重排模块，确保多轮追问下的检索召回率（Recall）依然处于极高水平。

### 🛠️ 核心重构 B：前端“两阶段状态提交保护”与格式适配

- **重构逻辑**：在 `app.py` 中，彻底颠覆了“用户输入 $\rightarrow$ 入库 $\rightarrow$ 传递”的传统时序。
- **机制流转**：

  1. **纯净历史提取**：当用户敲下回车时，首先通过 `for msg in current_session["history"]` 循环，**在当前输入尚未追加进历史列表前**，把过去的对话提取出来。
  2. **边界清洗与规整**：在提取过程中，通过特异性字段过滤掉 `system_welcome`（系统欢迎语），并将前端标准的 JSON 字典结构优雅地包装为 LangChain 原生识别的 `HumanMessage` 和 `AIMessage` 对象。
  3. **先流式渲染，成功后入库**：使用 `st.write_stream` 配合后端的迭代生成器 `iter_answer_deltas` 进行前端气泡的流式打字机渲染。**只有当流式生成顺利结束、完整拿到最终的** **`extracted_answer`** **时**，才通过 `.append()` 将 Assistant 的完整回复和 User 的当前提问同步提交进 `st.session_state` 的持久化历史中。

### 🛠️ 核心重构 C：流式参数链条的彻底打通

- **重构逻辑**：对 `test.py` 内部的三个核心流式输出函数（基于事件流的 `_yield_answer_deltas_from_events`、基于普通流的 `_yield_answer_deltas_from_stream` 以及对外暴露的 `iter_answer_deltas`）进行了彻底的形参对齐。
- **机制流转**：将 `chat_history: list | None = None` 作为标准一等公民参数逐层向下传递，并组装进最终触发 LangChain 的 `input_data` 字典中，彻底消除了静态类型检查器（Pyright）的报错，实现了参数在前后端的高速、稳定传导。

## 3. 改动带来的系统质变（Value）

通过本次改动，系统在架构体验和业务落地上面临着三重质变：

- **交互体验从“死板检索”升级为“专家会话”** ：
  由于引入了 `MessagesPlaceholder("chat_history")`，最终回答的大模型不仅在检索阶段能看懂历史，在**组织语言生成回答**阶段也能感知到前文提到的案例。例如，在用户没有重复提及的情况下，能主动将第一轮对话中的“房地产、大宗商品”等特异性压力情景融入第二轮的约束机制回答中，真正展现出“银行风险管理专家”在连续对话中逻辑层层递进、思路不被打断的深厚专业度。
- **工程稳定度与Token控制的全面闭环**：
  通过两阶段状态提交，彻底杜绝了模型拒答或网络崩溃时脏数据污染聊天窗口的顽疾。通过对系统欢迎语的过滤，在保持用户界面友好（第一句欢迎语对用户可见）的同时，避免了其进入大模型的多轮 Prompt 中占位，实现了极佳的 Token 经济性。
- **架构演进的“三位一体”核心结论**：
  改动完成后，本 RAG 系统的底层逻辑完美闭环：

  > **Rerank** 负责帮模型“找对真正重要的那张纸”；
  >
  > **Hierarchical** 负责帮模型“把这张纸背后的整本专业书的逻辑框架带进大脑”；
  >
  > **多轮历史感知** 则是帮模型“听懂人类不完整的连续追问，化身为能够随时调取整本书知识的私人风控顾问”。
