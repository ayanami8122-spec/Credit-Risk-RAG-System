从原本的全文本注入MVP，我们改进为了RAG（检索增强生成）系统，又引入 rerank 与 Hierarchical Retrieval。

## 一. 系统核心技术栈
目前我们的项目基于LangChain核心生态，在各组件之前有着明确的分工，实现了高内聚、低耦合的流水线架构：

```
[原始文档] -> [Hierarchical 切分] -> (父/子 Chunk 绑定) -> [Chroma 向量化存取]
                                                                ↓
[用户提问] -> [向量初筛 (bge-small)] -> [Rerank 交叉质检] -> [回溯父级上下文] -> [LLM 闭环生成]

```

### 1. 文档加载（Document Loading）
- **组件**：`TextLoader`
- **设计思考**：保持轻量化，完美保留 `.md` 格式的原始标记。放弃使用 `UnstructuredMarkdownLoader`，因为它会清洗掉 Markdown 的标点符号且较为重型、稳定性不够高。既然我们已经使用结构化的 Markdown 文档作为知识库，保持原始语意标记对后续的层级理解至关重要。

### 2. 文本切片与层级构建（Text Splitting & Hierarchical Structure）

- **组件**：`RecursiveCharacterTextSplitter` + **ParentDocumentRetriever（新增）**
- **设计思考**：传统的切片容易让信息断裂。我们引入了层级检索（Hierarchical Retrieval）机制，不再采用单一尺寸切片，而是建立“父-子”双层绑定关系：
- **子切片（Child Chunk）** ：设置较小的尺寸（如 `chunk_size=150`，`overlap=50`），用于保持极高的高频语义纯度，方便向量检索精准定位。
- **父切片（Parent Chunk）** ：设置较大的尺寸（如 `chunk_size=1200`），保留完整的上下文、金融理论框架或监管背景。
- **参数演进**：目前小型知识库子切片原本设为 150 到 300；在处理金融、法律类强逻辑文本时，父切片可拓展至 800 到 1200，确保大模型看得到完整的因果传导链条。

### 3. 向量化编码（Embeddings）

- **组件**：`BAAI/bge-small-zh-v1.5`（部署于本地的 Hugging Face 高性能中文向量模型）
- **底层逻辑**：把自然语言翻译为高维数字向量，利用数学公式（如余弦相似度）计算坐标距离，精准揪出最相关的知识。它赋予了系统跨语言的能力，使例如：“违约概率”与 “Probability of Default（PD）”这种不同语言但是意义相同的概念在数学空间中映射到几乎相同的坐标区域。
- **检索策略调整**：配合 Rerank 机制，我们将向量初筛的范围扩大，把 `vectorstore.as_retriever` 的参数调整为 `{"k": 15}`（粗筛出前 15 个卡片，表示15个子块，大致对应了3~4个父块），防止漏掉重要信息。

### 4. 向量数据库（Vector Store）

- **组件**：`Chroma`（`一个专门用来存这些“高维数字向量”的仓库`）
- **设计思考**：轻量级、支持本地持久化。它在后台只匹配数字（速度极快），一旦定位到距离最近的**子切片（Child Vector）** ，它不会直接把碎纸片喂给大模型，而是通过 Key-Value 映射，**向上捞出对应的“父切片文本（Parent Chunk）”**  再把这个文本喂给大模型。这彻底解决了传统向量数据库召回内容“断章取义”的顽疾。

### 5. **检索重排器（Reranker —— 新增核心）**

- **组件**：`BCELogger/bge-reranker-large`（或同类交叉注意力重排模型）
- **设计思考**：向量检索（Embedding）只管空间距离，很容易被字面相似度欺骗，捡芝麻丢西瓜。我们引入 **Rerank** 作为第二道过滤器，利用 Cross-Attention 机制让问题与初筛出的 10 个 Chunk 进行全文本逐字交互打分。它能像风控专家一样，跳出表面文字，识别出哪些 Chunk 才是真正包含“2008年金融危机、三角债、清收周期”等硬核理论背景的核心内容，并将最优质的 `k=3` 张卡片精准送渡给大模型。

### 6. LLM 与应用链组装（Chains）

- **组件**：`ChatOpenAI` + `create_stuff_documents_chain` / `create_retrieval_chain`
- **底层动作**：LangChain 将整个流水线打包。我们只需要调用一次调用 `rag_chain.invoke` ，底层就会自动完成：“向量粗筛 $\rightarrow$ Rerank 精选 $\rightarrow$ 父级上下文回溯 $\rightarrow$ 材料打包填充 $\rightarrow$ LLM 闭环问答”的全套复杂动作。
- **组件作用**：reate_stuff_documents_chain ：材料填充，作用是把找到的几张知识卡片文本直接打包、填充到{context}中去再把用户问题填充到{input}中去，最后组成一段完美、带有材料的prompt给到大模型，但它不知道去哪找材料，只能等其他工具喂给它； create_retrieval_chain 把检索器和刚刚的QA链连接起来
## 二、 LangChain
### 如果没有 LangChain / LlamaIndex 生态，手动实现这套 Advanced RAG 你要写多少代码？

1. **文件切片与绑定**：要自己用 Python 写代码去打开文本、手写逻辑去数系统字数进行切片。而且，由于引入了层级检索（Hierarchical），你得手写一套复杂的数据库（如 Redis 或 Local Dict）映射逻辑，自己去维护几百个“子切片 ID”和“父切片文本”之间的字典绑定关系。一旦切错一个 Token，整个父子映射直接报废。
2. **手写网络请求与本地加载**：你要自己写复杂的网络请求或本地 Transformers 管道去调 Hugging Face 的接口，小心翼翼地处理张量（Tensor）和拿回来的高维数组。
3. **自己写数学公式算矩阵**：你要自己研究怎么用数学公式写余弦相似度，在内存里用 NumPy 去对比这些上百维的数组，还要自己写排序算法（Top-K）把前 10 个粗筛结果捞出来。
4. **手写双重质检重排（Rerank）** ：捞出 Top-10 之后你还不能直接用，你还得自己写网络请求把这 10 个 Chunk 和 User Query 打包，发给本地或云端的 Rerank 模型，自己解析返回的评分 JSON，再写一遍排序算法，精准挑出前 3 个。
5. **极其恶心的字符串拼接**：你要手写极其复杂的字符串拼接，把 Rerank 挑出的那 3 个子切片对应的**父切片完整材料**和用户问题死板地拼好，再用 requests 库发给大模型，最后还要自己去小心翼翼地解析返回的 JSON 字典，处理各种网络超时和流式传输（Streaming）。

### 而有了 LangChain 生态：

它把这些业界的标准做法全部变成了标准类（Class）。

你需要什么，直接 `from langchain... import ...`，然后**像搭乐高一样**，用极其优雅的 **LCEL（LangChain Expression Language）表达式** 或者几行组件组装，把层级检索器（ParentDocumentRetriever）、Rerank 优化器（ContextualCompressionRetriever）、提示词、模型几行代码串起来。

项目在底层自动完成了  **“子空间粗筛** **$\rightarrow$** **交叉重排** **$\rightarrow$** **父级回溯** **$\rightarrow$** **自动打包** **$\rightarrow$** **LLM 闭环生成”**  的全部复杂动作，让项目直接具备了工业级的生产能力。
## 三、 创新与技术更新
### 1. 从“全文本盲目注入” 到 “高语义文本切片”

- **过去**：直接读取全文本，效率极低，极易触发 Token 上限，且由于文本过于冗长，大模型会出现“大海捞针（Needle in a Haystack）”式的记忆迷失。
- **现在**：通过 `RecursiveCharacterTextSplitter` 动态切分，按段落-句子-空格逐级降解，用滑动窗口保证语义不割裂。

### 2. 从“表面关键词匹配” 到 “本地向量化粗筛”

- **过去**：依赖字面关键词，需要大模型去读自然语言文本，容易受到同义词、缩写（如“违约总敞口”与“EAD”）拼写不同的干扰。
- **现在**：本地部署 `bge-small-zh-v1.5`，让计算机在大模型思考前先在数学高维空间进行快速筛选，大幅节省 Token 并提升响应速度。

### 3.  **从“局部信息复述” 到 “知识体系重构”（Rerank + Hierarchical）**
- **痛点**：前两版系统虽然能找到“发生了什么”，但由于内容扁平、碎片化，回答缺乏金融理论深度和监管框架（只懂经济差抵押物会跌，不懂巴塞尔协议的动态预期损失 EL 计量和顺周期资本缓冲）。
- **现在**：

  - **Rerank（重排）解决了“找得准”的问题**：它砍掉了向量检索带来的表面相关噪声，确保真正高价值、有深度的知识卡片排在最前面。
  - **Hierarchical（层级检索）解决了“讲得深”的问题**：它恢复了知识库的树状逻辑结构。模型不仅拿到了“叶子节点（案例）”，还顺着树干继承了“全局上下文（监管框架与风险计量公式）”。
  - **效果**：AI 的输出质量发生质变，角色成功从**自媒体科普解说员**升级为了**银行风险管理专家**。
