# 📝 RAG 项目全周期实战复盘与深度思考总结

## 一、 架构演进：从“全文本注入”到“三位一体”检索系统

在项目初期，我们意识到简单的全文本注入存在效率低、Token 浪费及信息断裂等问题。通过引入 LangChain 生态，我们将架构升级为高内聚、低耦合的流水线。

### 1. Hierarchical Retrieval（层级检索机制）

- **痛点**：传统固定尺寸切片容易导致语义断裂，大模型难以理解完整的上下文逻辑。
- **方案**：建立“父-子”双层绑定关系。

  - **子切片（Child Chunk）** ：设置较小尺寸（如 150-300 tokens），用于精准的向量空间检索定位。
  - **父切片（Parent Chunk）** ：设置较大尺寸（如 1200 tokens），在定位后通过 KV 映射回溯完整上下文，确保大模型看清因果传导链条。
- **选型**：使用 `TextLoader` 保持原始 Markdown 标记，放弃重型 Loader 以保障稳定性。

### 2. Rerank（交叉注意力重排）

- **设计思考**：向量检索仅关注空间距离，易受字面相似度欺骗。我们引入 **Flashrank** 作为第二道质检。
- **效果**：利用 Cross-Attention 机制对初筛出的 15 个 Chunk 进行逐字打分，识别出真正核心的内容（如金融危机背景、清收周期等），精选出最优 Top-3 喂给大模型。

## 二、 交互智能：多轮对话感知与上下文清洗

为了解决系统“听不懂人话”和“忘掉上文”的问题，我们在第三阶段引入了对话感知能力。

### 1. 历史感知检索器（History-Aware Retriever）

- **指代消解**：通过大模型预处理，将用户模糊的追问（如“它面临什么挑战？”）结合历史记录重写为独立饱满的 Query（如“量化 Downturn LGD 时面临什么挑战？”），极大提升了多轮对话下的召回率。

### 2. 纯净历史提取

- **策略**：在提交 Prompt 前特异性过滤掉 `system_welcome` 等噪声，并将前端 JSON 格式转化为 LangChain 原生的 `HumanMessage` 与 `AIMessage` 对象，杜绝无效 Token 浪费。

## 三、 工程落地：UI 性能优化与持久化管理

将 CLI 调试模式转为 Web 端（Streamlit）时，面临着严重的资源与状态管理挑战。

### 1. 生命周期管理（内存保护）

- **问题**：Streamlit 的“自上而下”运行机制会导致每次交互都重载模型，引发内存暴涨。
- **解决**：使用 `@st.cache_resource` 装饰器接管后端高维模型与数据库连接，确保整个运行期间资源仅初始化一次，实现前后端解耦。

### 2. SQLite 持久化与多会话逻辑

- **双表关联**：在 `database.py` 中通过 `chat_sessions` 与 `chat_messages` 建立外键级联，支持多会话留存。
- **焦点兜底**：在删除当前激活会话时，加入动态指针重置逻辑，确保 UI 永远有合法渲染视图，避免 `None` 指针崩溃。

### 3. 两阶段状态提交保护

- **机制**：采取“先流式渲染，成功后再入库”的时序。只有当 API 顺利返回完整答案且未触发异常时，才将对话存入数据库，有效避免了因网络抖动或风控拒答产生的脏数据入库。

## 四、 避坑指南：环境适配与控制流教训

在 Windows 环境下的开发过程暴露出许多底层兼容性问题。

### 1. 编码与信号处理

- **编码截断**：Windows 默认编码冲突会导致终端解析出“空回车”触发逻辑空转。解决：在程序入口强行重置 `sys.stdin` 为 `utf-8`。
- **退出信号拦截**：内层过宽的 `except Exception` 会误捕获 `Ctrl+C` 信号。解决：将 `KeyboardInterrupt` 提升至最外层循环作用域，确保程序能安全杀死进程。
- **文件锁冲突**：重命名目录时若后台服务未关，会触发 `EBUSY` 错误。需彻底释放 SQLite 句柄并关闭守护进程。

### 2. Prompt 工程的平衡术

- **过度对齐风险**：绝对化约束（如“必须完全根据知识库”）会导致模型在面对英文术语或微小偏差时拒答卡死。
- **引导胜过约束**：将指令改为“规则引导”，开启跨语言对齐支持，允许模型在保留底线的前提下进行专业语意扩充。

## 五、 总结启示：为什么选择 LangChain？

如果没有 LangChain 生态，手动实现上述层级切片绑定、余弦相似度计算、Cross-Attention 排序及复杂的字符串拼接将面临巨大的代码维护压力。通过 **LCEL（表达式语言）** ，我们像搭乐高一样组装组件，使项目具备了工业级的生产能力。

**核心感悟**：一个优秀的 RAG 系统不仅是算法的堆砌，更是对底层环境（编码）、控制流异常以及模型交互心理（Prompt 宽严度）的深度调优。
